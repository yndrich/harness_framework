# Architecture Decision Records

## 철학
출처 추적과 사람 검토 가능성을 속도보다 우선한다. 외부 의존성은 0개로 가져간다.
'마법처럼 정답을 주는 블랙박스'보다 '왜 이 값인지 추적 가능한 도구'를 만든다.

---

### ADR-001: 원본 XBRL 파싱 대신 companyfacts API 사용
**결정**: `data.sec.gov/api/xbrl/companyfacts/CIK##########.json` 한 번 호출로
한 기업의 모든 us-gaap 사실을 받는다. 공시별 XBRL 인스턴스 문서를 직접 파싱하지 않는다.
**이유**: companyfacts 는 이미 us-gaap 태그로 표준화돼 연도/분기별로 정리돼 있다.
3대 재무제표의 집계값은 전부 여기 있고, 파싱 코드·실패 지점이 사라진다.
**트레이드오프**: companyfacts 는 **차원(dimension) 데이터를 담지 않는다.** 부문별·지역별
매출, KPI 분해는 여기서 못 얻는다 → 별도 phase(inline-XBRL)로 미룬다 (ADR-007).

### ADR-002: SEC 단독 + 직접 표준화 (외부 유료 벤더 거부)
**결정**: 표준화·재작성 처리를 직접 한다. FMP/Sharadar/Intrinio 같은 정규화 벤더를 쓰지 않는다.
**이유**: 벤더는 모호한 매핑·재작성 판단을 '조용히' 내려 출처 추적을 가린다. 우리의 핵심
요구는 그 판단을 사람이 검토 가능하게 만드는 것이라 벤더의 편의가 오히려 요구와 충돌한다. SEC 는 무료·권위 출처.
**트레이드오프**: 매핑 레이어를 우리가 유지해야 한다(새 기업에서 GAP 나면 후보 태그 추가).

### ADR-003: canonical-map 표준화 레이어 (연도별 후보 태그 우선순위)
**결정**: `canonical_map.py` 가 '표준 항목 → 후보 us-gaap 태그 우선순위 리스트'를 정의.
정규화는 **연도마다** 위에서부터 첫 값을 채택한다.
**이유**: 같은 개념이 기업·연도마다 다른 태그를 쓰고 시간이 지나며 바뀐다(예: 매출 =
`RevenueFromContractWithCustomerExcludingAssessedTax` ↔ `Revenues` ↔ `SalesRevenueNet`).
연도별 해석이라야 태그 변경을 자동 흡수한다.
**트레이드오프**: 후보 둘 이상이 값을 가지면 모호 → 사람이 봐야 한다(ADR-006).

### ADR-004: 외부 의존성 0개 (stdlib only, XLSX 작성기 직접 구현)
**결정**: `urllib` 로 HTTP, 직접 만든 `xlsx.py` 로 엑셀. `requests`/`openpyxl` 미사용.
**이유**: 실행 환경에 pip/ensurepip 가 없어 설치가 불가능하다. 의존성 0이면 어디서든 바로 돈다.
**트레이드오프**: XLSX 작성기를 직접 유지(셀 스타일·병합·틀고정만 지원, 차트·피벗 없음).

### ADR-005: 재작성(restatement) = 최신 제출본 채택 + 플래그
**결정**: 같은 (항목, 연도)가 여러 공시에 나오면 가장 최근 `filed` 값을 채택하고,
값이 갈리면 `RESTATED` 로 표시(과거값 보존).
**이유**: companyfacts 는 같은 기간을 여러 공시(다른 accession)로 반환한다. 최신 = 현재 시점의 가장 정확한 값.
**트레이드오프**: '최초 보고값(as-originally-reported)' 뷰는 기본 제공 안 함(후속 토글 여지).

### ADR-006: 사람 검토 플래그 (AMBIGUOUS / RESTATED / GAP)
**결정**: 자동 판단이 애매한 칸을 색상 하이라이트 + 전용 Review 시트로 표면화한다.
**이유**: 자동화가 틀릴 수 있는 지점을 숨기지 않고 드러내는 것이 제품의 신뢰성과 직결된다.
**트레이드오프**: 플래그가 많으면 사용자가 일부 검토 노동을 한다(의도된 비용).

### ADR-007: 부문/지역/KPI 는 별도 phase 로 분리 (inline-XBRL)
**결정**: 차원 데이터는 MVP 에서 제외하고, 공시의 inline-XBRL 인스턴스를 파싱하는
phase `1-segments-kpi` 로 미룬다.
**이유**: 난이도·실패 지점이 3대 재무제표(ADR-001 경로)보다 훨씬 크다. 쉬운 80% 가치를 먼저 확보.
**트레이드오프**: 부문/지역/KPI 를 원하면 다음 phase 완료까지 기다려야 한다.
